# 欺诈类型与数据集字段映射

## 1. 说明

本文件用于回答“本课题研究的三类欺诈行为，在当前 `transaction_dataset.csv` 中分别对应哪些具体字段”这一问题。考虑到当前数据集是**地址级聚合统计数据**，而非原始交易图数据，本文件将字段分为三类：

- **核心字段**：与该欺诈类型业务逻辑直接相关，可作为主要分析依据。
- **辅助字段**：可增强解释力，但不是最直接的识别信号。
- **当前数据集无法表达的关键信息**：需要原始交易明细、交易图、合约代码或外部标签才能获得。

为便于论文书写，本文件中的字段名采用**去除首尾空格后的标准化写法**。例如：

- `Total ERC20 tnxs` 对应原始 CSV 中的 ` Total ERC20 tnxs`
- `ERC20 total Ether received` 对应原始 CSV 中的 ` ERC20 total Ether received`
- `max value received` 对应原始 CSV 中的 `max value received `

## 2. 钓鱼诈骗与资金清洗

### 2.1 核心字段

| 特征维度 | 字段名 | 对应理由 |
| --- | --- | --- |
| 生命周期 | `Time Diff between first and last (Mins)` | 钓鱼地址通常短期活跃后迅速废弃，生命周期明显偏短。 |
| 发送节奏 | `Avg min between sent tnx` | 攻击者往往用脚本快速转出资产，发送间隔很短。 |
| 净留存 | `total ether balance` | 受害资金被快速清空后，地址中通常不保留明显净余额。 |

### 2.2 辅助字段

| 特征维度 | 字段名 | 用途说明 |
| --- | --- | --- |
| 接收节奏 | `Avg min between received tnx` | 观察被害资金或归集资金在时间上的集中程度。 |
| 交易频次 | `Sent tnx` | 判断是否存在明显的快速出金行为。 |
| 交易频次 | `Received Tnx` | 判断是否存在集中收款后迅速转出的模式。 |
| 总交易量 | `total transactions (including tnx to create contract` | 描述整体交易活跃度。 |
| 交互广度 | `Unique Received From Addresses` | 判断是否从多个地址快速归集资金。 |
| 交互广度 | `Unique Sent To Addresses` | 判断是否向多个地址扩散或分流资金。 |
| 收款金额 | `min value received` | 观察小额试探性转账或多笔归集。 |
| 收款金额 | `max value received ` | 观察是否出现大额受害资金流入。 |
| 收款金额 | `avg val received` | 用于描述平均收款规模。 |
| 付款金额 | `min val sent` | 观察是否存在大量小额试探转出。 |
| 付款金额 | `max val sent` | 观察是否存在单笔大额清空。 |
| 付款金额 | `avg val sent` | 描述平均出金规模。 |
| ETH 流量 | `total Ether sent` | 描述累计转出规模。 |
| ETH 流量 | `total ether received` | 描述累计流入规模。 |

### 2.3 可选补充字段

若考虑“钓鱼后通过代币授权转移资产”的场景，还可补充观察：

- `Total ERC20 tnxs`
- `ERC20 total Ether received`
- `ERC20 total ether sent`
- `ERC20 uniq sent addr`
- `ERC20 uniq rec addr`
- `ERC20 most sent token type`
- `ERC20_most_rec_token_type`

这些字段对“代币类钓鱼”具有补充解释价值，但不是当前课题中钓鱼与资金清洗的最强主特征。

### 2.4 当前数据集无法直接表达的信息

- 资金是否转入混币器或交易所
- 是否存在地址投毒关系
- 是否存在恶意授权调用
- 多跳资金转移路径

因此，本数据集更适合描述“短生命周期 + 快速清空 + 低留存”的行为后果，而不是完整恢复钓鱼攻击链条。

## 3. 庞氏骗局

### 3.1 核心字段

| 特征维度 | 字段名 | 对应理由 |
| --- | --- | --- |
| 多源入金 | `Unique Received From Addresses` | 庞氏骗局通常吸引大量不同地址持续入金。 |
| 接收频次 | `Received Tnx` | 收款交易频次高，反映持续吸金。 |
| 收发对比 | `Sent tnx` | 用于与 `Received Tnx` 对比，判断“收款显著多于付款”的特征。 |
| 金额模式 | `avg val received` | 观察是否存在固定投资额度或明显金额聚类。 |

### 3.2 辅助字段

| 特征维度 | 字段名 | 用途说明 |
| --- | --- | --- |
| 生命周期 | `Time Diff between first and last (Mins)` | 庞氏骗局一般较钓鱼地址存续更久，可辅助区分。 |
| 总交易量 | `total transactions (including tnx to create contract` | 反映整体参与度和活跃度。 |
| 出金广度 | `Unique Sent To Addresses` | 判断是否向多个参与者进行返利或分发。 |
| 累计入金 | `total ether received` | 反映资金池规模。 |
| 累计出金 | `total Ether sent` | 观察返利或提现规模。 |
| 资金净额 | `total ether balance` | 观察地址净沉淀情况。 |
| 收款金额 | `min value received` | 分析最小投资门槛。 |
| 收款金额 | `max value received ` | 分析是否存在大额参与者。 |
| 付款金额 | `min val sent` | 反映最小返利或提现规模。 |
| 付款金额 | `max val sent` | 反映最大返利或资金转出规模。 |
| 平均付款 | `avg val sent` | 用于和平均入金额度联合分析。 |

### 3.3 合约型庞氏的补充字段

如果考虑以智能合约形式运行的庞氏骗局，还可以补充：

- `Number of Created Contracts`
- `total ether sent contracts`
- `min value sent to contract`
- `max val sent to contract`
- `avg value sent to contract`

这些字段可用于分析地址是否参与了合约创建或与合约有明显资金交互，但在当前数据集中仍然只是间接信号。

### 3.4 当前数据集无法直接表达的信息

- 星状或树状资金传播结构的完整图形
- 前后参与者之间的资金传递链
- 合约内部“分红 / 返利 / 层级分发”的代码逻辑

因此，对于庞氏骗局，本数据集更适合从“多源收款 + 高频入金 + 金额模式”角度做行为分析，而不是直接证明其完整层级结构。

## 4. 发币骗局与跑路盘

### 4.1 核心字段

| 特征维度 | 字段名 | 对应理由 |
| --- | --- | --- |
| 合约部署 | `Number of Created Contracts` | 发币骗局和跑路盘往往伴随代币合约或相关合约的创建。 |
| ERC-20 活跃度 | `Total ERC20 tnxs` | 代币交易越活跃，越可能存在营销、吸筹或异常代币流转。 |
| ERC-20 资金规模 | `ERC20 total Ether received` | 反映地址接收代币资产的总体规模。 |

### 4.2 辅助字段

| 特征维度 | 字段名 | 用途说明 |
| --- | --- | --- |
| ERC-20 发送规模 | `ERC20 total ether sent` | 观察代币是否被持续转出或转移。 |
| ERC-20 合约发送 | `ERC20 total Ether sent contract` | 描述与合约层的代币转移关系。 |
| ERC-20 网络广度 | `ERC20 uniq sent addr` | 观察向多少不同地址发送代币。 |
| ERC-20 网络广度 | `ERC20 uniq rec addr` | 观察从多少不同地址接收代币。 |
| ERC-20 合约交互 | `ERC20 uniq rec contract addr` | 反映与代币合约相关地址的交互范围。 |
| ERC-20 收款金额 | `ERC20 min val rec` | 分析最小代币接收规模。 |
| ERC-20 收款金额 | `ERC20 max val rec` | 分析最大代币接收规模。 |
| ERC-20 收款金额 | `ERC20 avg val rec` | 分析平均代币接收规模。 |
| ERC-20 付款金额 | `ERC20 min val sent` | 分析最小代币发送规模。 |
| ERC-20 付款金额 | `ERC20 max val sent` | 分析最大代币发送规模。 |
| ERC-20 付款金额 | `ERC20 avg val sent` | 分析平均代币发送规模。 |
| 代币多样性 | `ERC20 uniq sent token name` | 判断是否涉及多个代币。 |
| 代币多样性 | `ERC20 uniq rec token name` | 分析接收代币类型数量。 |
| 代币类型 | `ERC20 most sent token type` | 识别最常发送的代币类型。 |
| 代币类型 | `ERC20_most_rec_token_type` | 识别最常接收的代币类型。 |
| 合约资金交互 | `total ether sent contracts` | 判断 ETH 是否被持续发送至合约。 |
| 合约资金交互 | `min value sent to contract` | 分析向合约发送 ETH 的金额下限。 |
| 合约资金交互 | `max val sent to contract` | 分析向合约发送 ETH 的金额上限。 |
| 合约资金交互 | `avg value sent to contract` | 描述平均向合约投入的资金。 |

### 4.3 理论相关但当前数据中无有效区分能力的字段

以下字段从业务上与 ERC-20 或合约行为相关，但在当前数据集中几乎全部为 0，暂时不适合作为有效分析特征：

- `ERC20 avg time between sent tnx`
- `ERC20 avg time between rec tnx`
- `ERC20 avg time between rec 2 tnx`
- `ERC20 avg time between contract tnx`
- `ERC20 min val sent contract`
- `ERC20 max val sent contract`
- `ERC20 avg val sent contract`

### 4.4 当前数据集无法直接表达的信息

- 合约源码中是否存在后门、黑名单、税率控制、隐藏铸币等逻辑
- 是否发生了撤池、抽走流动性、锁仓解除等事件
- 代币价格走势、交易深度和市值变化
- 与去中心化交易所流动性池的直接关系

因此，本数据集更适合分析“合约创建 + ERC-20 交互活跃 + 代币异常流转”的行为表征，而不能单独完成对 rug pull 机制的完全验证。

## 5. 三类欺诈类型的字段覆盖情况总结

| 欺诈类型 | 最强字段簇 | 代表字段 |
| --- | --- | --- |
| 钓鱼诈骗与资金清洗 | 时间节奏 + 净余额 + ETH 流转 | `Time Diff between first and last (Mins)`、`Avg min between sent tnx`、`total ether balance` |
| 庞氏骗局 | 多源入金 + 接收频次 + 金额聚类 | `Unique Received From Addresses`、`Received Tnx`、`avg val received` |
| 发币骗局与跑路盘 | 合约创建 + ERC-20 行为 + 代币类型 | `Number of Created Contracts`、`Total ERC20 tnxs`、`ERC20 total Ether received` |

## 6. 结论

当前数据集可以较好支撑本课题对三类欺诈类型进行字段层面的映射与行为分析。其中：

- 钓鱼诈骗与资金清洗更依赖时间和资金净流出特征；
- 庞氏骗局更依赖多源入金和接收行为特征；
- 发币骗局与跑路盘更依赖合约与 ERC-20 特征。

但需要强调的是，这些映射属于“行为特征映射”，而非“业务标签直接给定”。因此，后续若要进行严格的三类欺诈分类，还需要补充人工规则标注、外部标签或原始交易图信息。
