# 数据清洗方案

## 1. 清洗目标

本方案面向 `transaction_dataset.csv`，目标不是修改原始数据事实，而是将当前数据整理为：

- 字段命名统一、语义明确；
- 重复记录和冲突记录可解释；
- 缺失值和异常值有一致处理规则；
- 可以直接服务于后续的特征分析、模型训练和论文撰写。

建议保留原始文件不变，另行生成清洗后的分析版本。

## 2. 清洗原则

### 2.1 原始数据不覆盖

- 保留原始 `transaction_dataset.csv` 作为只读底稿。
- 所有清洗结果应输出为新文件，例如 `transaction_dataset_cleaned.csv`。
- 同时保留一份字段映射和清洗日志，确保论文过程可追溯。

### 2.2 先统一语义，再处理数值

在本数据集中，最大的风险不是简单缺失值，而是：

- 列名不规范；
- 字段语义不清；
- 同一空值被编码成多种形式；
- 个别字段可能是派生值而非原始业务值。

因此，清洗工作应先解决“字段是否可信、如何解释”的问题，再进行常规填补和建模准备。

## 3. 具体清洗步骤

### 3.1 字段命名标准化

#### 处理内容

- 去掉所有字段名首尾空格。
- 修正截断字段名：
  - `total transactions (including tnx to create contract`
- 对命名不清的字段单独标注：
  - `ERC20 uniq sent addr.1`

#### 处理建议

- 将所有字段名统一整理为“无前后空格、无隐式重复、语义完整”的格式。
- 对 `ERC20 uniq sent addr.1` 不要直接主观改名，建议：
  1. 优先回查原始数据来源或原论文说明；
  2. 若无法确认含义，则在基线分析中暂时剔除；
  3. 在论文中注明“字段语义不明，未纳入核心分析”。

### 3.2 标识字段处理

#### 处理对象

- `Unnamed: 0`
- `Index`
- `Address`

#### 处理建议

- `Unnamed: 0`：作为纯行号字段，直接删除。
- `Index`：非唯一、语义不稳定，不作为分析特征，建议删除。
- `Address`：保留作为样本追踪主键，但不参与模型训练。
- `FLAG`：保留为标签字段，不参与清洗删除。

### 3.3 重复地址处理

#### 当前问题

- 共有 25 个重复地址，对应 50 条记录。
- 大部分重复地址的数值特征完全一致。
- 7 个重复地址只在 ERC-20 文本字段上出现差异。

#### 处理建议

1. 对于**完全重复**的地址记录，只保留 1 条。
2. 对于**数值一致、文本冲突**的地址记录：
   - 数值字段保留一份即可；
   - 文本字段可按以下顺序处理：
     - 优先保留非空值；
     - 若存在多个非空但不同的值，可记为众数；
     - 若仍无法确定，则统一标记为 `AmbiguousTokenType`。
3. 保留一份重复处理日志，记录被合并的地址及其差异字段。

### 3.4 缺失值处理

#### 3.4.1 ERC-20 数值字段缺失

##### 当前问题

ERC-20 相关数值字段普遍存在 829 个缺失值，且缺失比例完全一致。

##### 处理建议

- 将这类缺失优先解释为“该地址未发生 ERC-20 相关行为”。
- 对应数值字段可统一填充为 0。
- 同时建议新增一个指示字段：
  - `has_erc20_activity`

该字段用于区分“原本就是 0”与“由于未发生 ERC-20 行为而填补为 0”的情况，便于后续解释。

#### 3.4.2 ERC-20 文本字段缺失

##### 当前问题

文本字段 `ERC20 most sent token type` 和 `ERC20_most_rec_token_type` 中同时存在：

- `NaN`
- `'0'`
- `' '`

##### 处理建议

- 先执行字符串去空格处理。
- 将 `NaN`、`'0'`、空字符串统一编码为同一个类别，例如：
  - `NoTokenInfo`
- 对剩余正常文本值再进行后续编码。

这样可以避免模型或统计分析将“缺失”“0”“空格”误判为三个不同类别。

### 3.5 异常值与语义异常处理

#### 3.5.1 `total ether balance` 的负值问题

##### 当前问题

`total ether balance` 出现 978 个负值，而真实账户余额理论上不应为负。

##### 处理建议

不要直接将负值截断为 0，因为这会破坏原始信息。更合理的做法是：

1. 将该字段解释为“净 ETH 流量”而非“实时账户余额”；
2. 在论文中明确说明该字段更接近：
   - `total ether received - total Ether sent - total ether sent contracts`
3. 如果后续需要提高语义准确性，可新增一个更明确的字段名，例如：
   - `net_eth_flow`
4. 原字段可保留，但应在分析时加注释说明其业务含义。

#### 3.5.2 极低信息量字段

##### 当前问题

多列数值字段在全表恒为 0 或几乎恒为 0。

##### 处理建议

对于以下恒定列，可直接从建模特征中剔除：

- `ERC20 avg time between sent tnx`
- `ERC20 avg time between rec tnx`
- `ERC20 avg time between rec 2 tnx`
- `ERC20 avg time between contract tnx`
- `ERC20 min val sent contract`
- `ERC20 max val sent contract`
- `ERC20 avg val sent contract`

对于以下极低信息量列，建议先做业务评估后再决定是否保留：

- `total ether sent contracts`
- `min value sent to contract`
- `max val sent to contract`
- `avg value sent to contract`
- `ERC20 uniq sent addr.1`

若这些字段在最终样本中几乎无变化，可在基线模型中暂时删除。

### 3.6 冗余字段处理

#### 当前问题

部分字段属于由其他字段计算得到的派生字段，例如：

- `total transactions (including tnx to create contract)`  
  可由 `Sent tnx + Received Tnx + Number of Created Contracts` 得到。
- `total ether balance`  
  近似可由 `total ether received - total Ether sent - total ether sent contracts` 得到。

#### 处理建议

- 在统计描述阶段，这些字段可以保留，因为它们便于解释。
- 在模型训练阶段，建议进行冗余检验，避免多重共线性。
- 若建立基线模型，可优先删除完全可推导的聚合字段，只保留基础字段。

### 3.7 数据类型与文本格式统一

#### 处理建议

- 所有数值列统一转为标准数值类型。
- 所有文本列统一进行：
  - 去前后空格；
  - 大小写和空值表达统一；
  - 异常符号检查。

对于文本类别字段，若后续需要参与模型训练，可根据实验阶段采用：

- 先删除；
- 频次编码；
- One-Hot 编码；
- 仅保留高频类别，其他合并为 `OtherToken`。

## 4. 面向论文写作的推荐清洗结果

建议将清洗结果整理成以下三类产物：

1. **清洗后数据文件**
   - `transaction_dataset_cleaned.csv`
2. **字段字典**
   - 原字段名
   - 标准字段名
   - 中文含义
   - 是否保留
   - 清洗方式
3. **清洗日志**
   - 删除了哪些字段
   - 合并了哪些重复地址
   - 哪些字段填充了 0
   - 哪些字段被标记为语义不明确

这样做的好处是，后续论文在“数据预处理”章节中可以直接说明每一步的依据，增强研究过程的规范性和可复现性。

## 5. 推荐的最终清洗顺序

为保证过程清晰，建议按以下顺序执行：

1. 备份原始数据
2. 标准化列名
3. 删除无意义标识字段
4. 处理重复地址
5. 统一文本空值表达
6. 处理 ERC-20 数值缺失
7. 识别并剔除恒定列
8. 评估低信息量列和冗余列
9. 输出清洗后数据、字段字典和日志

## 6. 结论

本数据集的清洗重点不在于“大量随机缺失值”，而在于“字段命名不规范、地址重复、文本编码混乱、语义不清和特征冗余”。只要按照本方案完成规范化处理，数据集就能够较稳定地支撑后续的欺诈类型分析、特征工程和毕业论文写作。
