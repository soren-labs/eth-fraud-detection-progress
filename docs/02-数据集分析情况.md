# 数据集分析情况

## 1. 数据集总体概况

当前课题的核心数据文件为 `transaction_dataset.csv`，属于**地址级聚合统计数据集**。经实际检查后，数据集具有以下基本特征：

- 总样本数：9841
- 总字段数：51
- 标签字段：`FLAG`
- 正常样本数：7662
- 欺诈样本数：2179

`FLAG` 只有 `0` 和 `1` 两类，因此该数据集本质上是“正常地址 vs 欺诈地址”的二分类数据，而不是直接标注了“钓鱼 / 庞氏 / 发币跑路”的三分类数据。

## 2. 数据字段构成

结合字段含义与已有说明文档，数据集可以大致分为以下几类：

1. 标识字段：`Unnamed: 0`、`Index`、`Address`
2. 标签字段：`FLAG`
3. 时间类字段：如 `Avg min between sent tnx`、`Time Diff between first and last (Mins)`
4. 交易频次与网络类字段：如 `Sent tnx`、`Received Tnx`、`Unique Received From Addresses`
5. ETH 金额统计类字段：如 `total Ether sent`、`total ether received`、`total ether balance`
6. 合约交互类字段：如 `Number of Created Contracts`、`total ether sent contracts`
7. ERC-20 相关字段：如 `Total ERC20 tnxs`、`ERC20 total Ether received`、`ERC20 most sent token type`

整体上，这是一份结构较完整的行为画像型数据集，适合做欺诈检测和特征分析；但在字段规范性和数据一致性方面存在较明显问题。

## 3. 缺失信息分析

### 3.1 缺失值总体情况

经过统计，存在缺失值的字段共有 25 个，缺失主要集中在 ERC-20 相关字段中。

其中：

- 一整组 ERC-20 数值字段均有 829 个缺失值，占总样本的 8.42%。
- `ERC20 most sent token type` 缺失 2697 个，占 27.41%。
- `ERC20_most_rec_token_type` 缺失 871 个，占 8.85%。

### 3.2 缺失值的业务含义

当前缺失值并不完全等同于“数据采集失败”，其中相当一部分更可能表示：

- 该地址没有发生 ERC-20 代币交易；
- 该地址没有合约层面的相关行为；
- 某些文本字段没有可识别的主要代币类型。

因此，这类缺失不能简单按随机缺失处理，而应结合字段业务语义进行判定。

### 3.3 主要缺失字段

下列字段均存在 829 个缺失值，缺失比例一致，说明它们很可能属于同一批 ERC-20 事件未发生而导致的“结构性缺失”：

- `Total ERC20 tnxs`
- `ERC20 total Ether received`
- `ERC20 total ether sent`
- `ERC20 total Ether sent contract`
- `ERC20 uniq sent addr`
- `ERC20 uniq rec addr`
- `ERC20 uniq sent addr.1`
- `ERC20 uniq rec contract addr`
- `ERC20 avg time between sent tnx`
- `ERC20 avg time between rec tnx`
- `ERC20 avg time between rec 2 tnx`
- `ERC20 avg time between contract tnx`
- `ERC20 min val rec`
- `ERC20 max val rec`
- `ERC20 avg val rec`
- `ERC20 min val sent`
- `ERC20 max val sent`
- `ERC20 avg val sent`
- `ERC20 min val sent contract`
- `ERC20 max val sent contract`
- `ERC20 avg val sent contract`
- `ERC20 uniq sent token name`
- `ERC20 uniq rec token name`

## 4. 数据中的混乱信息与可疑点

### 4.1 列名不规范

数据集列名存在明显的命名不统一问题：

- `max value received ` 带有尾部空格。
- 大量 ERC-20 字段带有前导空格，如 ` Total ERC20 tnxs`。
- `total transactions (including tnx to create contract` 疑似字段名被截断，缺少闭合部分。
- `ERC20 uniq sent addr.1` 明显是自动追加的重名字段格式，但原始含义不清晰。

这会导致后续代码读取、字段映射、论文表述和模型训练时出现不一致。

### 4.2 标识字段语义混乱

- `Unnamed: 0` 是严格递增的行号字段，可视为无意义索引。
- `Index` 不是唯一值，在 `FLAG=0` 样本中范围为 1 到 4729，在 `FLAG=1` 样本中范围为 1 到 2179，说明它更像是“分组内部编号”，而不是全局主键。
- `Address` 理论上应当是地址唯一标识，但实际存在重复。

因此，这三个字段不能同时被视为可靠主键。

### 4.3 地址重复问题

经统计：

- 完全重复行数量：0
- 重复地址涉及的行数：50
- 重复地址个数：25

其中：

- 大多数重复地址对应的数值特征完全一致；
- 7 个重复地址存在字段冲突；
- 冲突基本集中在 `ERC20 most sent token type` 和 `ERC20_most_rec_token_type` 两个文本字段。

这说明数据集在地址级聚合后仍存在重复记录或合并不一致的问题。

### 4.4 文本字段编码混乱

`ERC20 most sent token type` 和 `ERC20_most_rec_token_type` 两个字段中，同时存在以下几种“空值表达”：

- 真正的缺失值 `NaN`
- 字符串 `'0'`
- 仅包含空格的字符串 `' '`

其中：

- `ERC20 most sent token type` 中，“缺失 / 0 / 空白”合计 8287 条，占 84.21%。
- `ERC20_most_rec_token_type` 中，“缺失 / 0 / 空白”合计 5291 条，占 53.76%。

这说明文本字段的编码规则并不统一，直接用于建模会引入明显噪声。

### 4.5 存在恒定列与极低信息量列

以下数值字段在全表中恒为 0，实际上不包含可区分信息：

- `ERC20 avg time between sent tnx`
- `ERC20 avg time between rec tnx`
- `ERC20 avg time between rec 2 tnx`
- `ERC20 avg time between contract tnx`
- `ERC20 min val sent contract`
- `ERC20 max val sent contract`
- `ERC20 avg val sent contract`

此外，一些字段虽不完全恒定，但绝大多数取值都为 0，例如：

- `total ether sent contracts` 有 99.97% 的样本为 0；
- `min value sent to contract` 只有 3 个不同取值；
- `ERC20 uniq sent addr.1` 只有 4 个不同取值。

这类字段的信息增益可能很低，且需要重新判断业务意义。

### 4.6 存在语义可疑字段

`total ether balance` 出现了 978 个负值，占 9.94%。从真实链上账户余额的角度看，地址余额不可能为负，因此该字段很可能不是严格意义上的“当前余额”，而更像是：

- 净流入减净流出后的统计结果；
- 或者某种聚合窗口下的收支差值。

如果论文中直接将其解释为“账户余额”，会存在语义风险。

### 4.7 存在冗余和派生字段

经检查发现：

- `total transactions (including tnx to create contract` 完全等于  
  `Sent tnx + Received Tnx + Number of Created Contracts`
- `total ether balance` 近似等于  
  `total ether received - total Ether sent - total ether sent contracts`

这说明数据集中包含明显的派生汇总字段，后续建模时需要注意多重共线性和重复信息问题。

## 5. 数据集现阶段可得出的结论

### 5.1 优势

- 数据规模足够支持毕业设计层面的统计分析和模型实验。
- 涵盖时间、频次、金额、合约、ERC-20 多类特征，特征维度较丰富。
- 与课题选定的三类欺诈行为存在较好的特征对应关系。

### 5.2 局限

- 数据标签仅区分“正常 / 欺诈”，没有直接标注三种具体欺诈类型。
- 数据集不是交易图或原始时序明细，无法完整还原资金路径。
- 存在字段命名不规范、重复地址、文本编码混乱、恒定列和语义不清字段。

### 5.3 结论

该数据集可以作为本课题的主数据基础，但在正式建模和论文写作前，必须先完成字段规范化、重复记录处理、ERC-20 缺失统一、文本值归一化和冗余字段梳理等预处理工作。否则会影响后续特征分析、模型解释与论文表述的一致性。
