# 基于以太坊交易行为的数据特征总结报告

### 1. 数据集概况与目标变量

本研究所采用的数据集共包含 **9841** 条以太坊地址级别的聚合统计数据，合计 **51** 个数据维度。其中，目标变量（Label）为 `FLAG` 字段：

- **FLAG = 0**：代表正常交易地址，共计 7662 条样本。

- **FLAG = 1**：代表被标记为欺诈的交易地址（如钓鱼、资金盘等），共计 2179 条样本。
  数据呈现出典型的正负样本不平衡分布（约 3.5 : 1），将在后续建模中通过类别权重调整等策略进行处理。

### 2. 需剔除的无效与标识字段

在进入机器学习模型训练前，需剔除以下与交易行为本身无关、或不具备泛化能力的标识类字段：

- `Unnamed: 0` / `Index`：数据集原有的行号与索引字段。

- `Address`：以太坊账户的哈希地址。该字段为唯一标识符，保留以用于结果追溯与报告生成，但不参与模型训练。

### 3. 核心特征分类体系（参与模型训练）

经过清洗与梳理，参与模型训练的特征可严谨地划分为以下 **五大维度**：

#### 3.1 时间与存续特征 (Time-based Features)

反映地址的活跃生命周期与交易时间节奏。欺诈账户通常具有“短平快”的特征（即短期内高频交易后迅速废弃）。

- `Time Diff between first and last (Mins)`：地址从首笔交易到末笔交易的存活时间差（分钟）。

- `Avg min between sent tnx`：发送交易之间的平均时间间隔（分钟）。

- `Avg min between received tnx`：接收交易之间的平均时间间隔（分钟）。

#### 3.2 交易频次与网络特征 (Frequency & Network Features)

反映地址在以太坊网络中的活跃度以及交互的广度。欺诈节点往往与大量不同的受害者地址产生星状网络连接。

- `Sent tnx` / `Received Tnx`：作为发送方/接收方的历史交易总笔数。

- `total transactions`：总交易次数（含合约创建）。

- `Unique Received From Addresses` / `Unique Sent To Addresses`：交互过的独立发送方/接收方地址去重总数。

#### 3.3 基础资金金额统计特征 (ETH Amount Statistical Features)

反映以太坊原生代币（ETH）的资金流转规模。欺诈账户通常表现为资金的快速聚拢与分散转移，且极少在账户中留存高额余额。

- `total ether received` / `total Ether sent`：历史累计接收 / 发送的 ETH 总量。

- `total ether balance`：该地址当前的 ETH 余额。

- `min / max / avg value received`：单笔接收 ETH 的最小 / 最大 / 平均金额。

- `min / max / avg val sent`：单笔发送 ETH 的最小 / 最大 / 平均金额。

#### 3.4 智能合约交互特征 (Smart Contract Interaction Features)

反映地址与以太坊智能合约底层的交互行为。由于许多资金盘和钓鱼攻击依赖特定的恶意合约运行，该维度特征具有强业务指向性。

- `Number of Created Contracts`：该地址历史上创建的智能合约总数。

- `total ether sent contracts`：累计向智能合约发送的 ETH 总量。

- `min / max / avg value sent to contract`：单笔向合约发送资金的最小 / 最大 / 平均金额。

#### 3.5 ERC-20 代币特征 (ERC-20 Token Features)

以太坊生态中大量欺诈依托于虚假或特定的 ERC-20 代币发行。此类特征是对原生 ETH 特征的有力补充。（注：该类特征存在自然缺失值，代表该地址未发生过代币交易，需在预处理中填充为 0）。

- **频次与时间类**：`Total ERC20 tnxs` (总交易数), `ERC20 avg time between sent/rec tnx` (代币收发时间间隔) 等。

- **资金类**：`ERC20 total Ether received/sent` (代币收发总量), `ERC20 min/max/avg val rec/sent` 等。

- **网络类**：`ERC20 uniq sent/rec addr` (独立的代币收发交互地址数)。

- **多样性类**：`ERC20 uniq sent/rec token name` (涉及的独立代币种类数量)。

### 4. 文本类别字段（待特殊处理）

数据集中包含两个非数值型的分类字段：

- `ERC20 most sent token type`：发送最频繁的代币类型（文本形式）。

- `ERC20_most_rec_token_type`：接收最频繁的代币类型（文本形式）。 **处理建议：** 考虑到类别可能极为庞杂且易产生长尾效应，在基线模型中建议先做剔除；若后续需要提点模型精度，可考虑将其转换为频次编码（Frequency Encoding）或进行 One-Hot 编码。
